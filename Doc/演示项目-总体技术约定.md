# 演示项目 — 总体技术约定

> 版本：v1.0  
> 日期：2026-06-26  
> 适用范围：区域监控、预警研判、处置反馈、数据分析

---

## 1. 核心原则

### 1.1 Web 侧代码是「真实的」

| 要求 | 说明 |
|------|------|
| 统一 API 层 | 所有模块通过 `src/lib/api-client.ts` 发 `fetch`，禁止在组件里硬编码 Mock 数据 |
| Mock 仅替换传输层 | 数据来自 `Mock/` 目录，由 Vite 插件在开发/预览时拦截返回 |
| 业务逻辑在前端 | 运动计算、区域判定、状态机、模块联动在 `src/features/**` 实现 |
| POST 也走 Mock | 写操作同样 `fetch` + Mock 响应，前端根据响应更新 Store |

### 1.2 无真实后端

- 不部署 API 服务
- 不使用 WebSocket
- 动态位置由前端 `requestAnimationFrame` 根据运动配置实时计算

### 1.3 时间显示

- 状态栏时钟使用**浏览器本地实时时间**（`new Date()`），精确到秒
- 监控记录 / 行为时间线中的历史时间点使用**相对演示开始的模拟时间**，须早于状态栏当前时刻

## 2. Mock 目录规范

### 2.1 目录结构

Mock 文件路径与 API URL **一一对应**，根目录为项目下的 `Mock/`：

```
Mock/
└── api/
    └── v1/
        ├── monitoring/
        │   ├── scene.json                 # GET  /api/v1/monitoring/scene
        │   ├── targets.json               # GET  /api/v1/monitoring/targets
        │   └── demo/
        │       ├── start.post.json        # POST /api/v1/monitoring/demo/start
        │       ├── start-all.post.json    # POST /api/v1/monitoring/demo/start-all
        │       ├── reset.post.json        # POST /api/v1/monitoring/demo/reset
        │       └── timing.post.json       # POST /api/v1/monitoring/demo/timing
        ├── judgment/
        │   ├── alert.json                 # GET  /api/v1/judgment/alert      (U-004)
        │   ├── unknown.json               # GET  /api/v1/judgment/unknown
        │   └── warning.json               # GET  /api/v1/judgment/warning    (U-003)
        └── analytics/
            └── overview.json              # GET  /api/v1/analytics/overview
```

### 2.2 文件命名规则

| HTTP 方法 | 文件命名 |
|-----------|----------|
| GET | `{path}.json`，如 `scene.json` |
| POST / PATCH / PUT / DELETE | `{path}.{method}.json`，如 `start.post.json` |

URL `/api/v1/monitoring/demo/start` + `POST` → `Mock/api/v1/monitoring/demo/start.post.json`

### 2.3 响应包裹格式

Mock 文件**只存 `data` 字段内容**；Vite 插件统一包装：

```json
{
  "code": 0,
  "message": "ok",
  "data": { /* 来自 Mock 文件 */ }
}
```

错误示例（文件不存在时由插件生成）：

```json
{
  "code": 40401,
  "message": "mock not found: /api/v1/monitoring/xxx",
  "data": null
}
```

### 2.4 Vite 插件职责（`vite-plugin-mock-api`）

1. 拦截 `server.middlewares` 中 `req.url` 以 `/api/` 开头的请求
2. 解析 method + path，映射到 `Mock/api/...` 文件
3. 读取 JSON，包装后 `res.end(JSON.stringify(...))`
4. `pnpm dev` 与 `pnpm preview` 均生效

前端无需感知 Mock 存在：

```typescript
// src/lib/api-client.ts
export async function apiGet<T>(path: string): Promise<T> {
  const res = await fetch(path);
  const body = await res.json();
  if (body.code !== 0) throw new Error(body.message);
  return body.data as T;
}
```

## 3. 演示分段与并行策略

三场演示**可独立触发、可同时并行**。通过**不同到核心耗时**自然形成先后节奏，无需互斥锁：

| 优先级 | 分段 `segment` | 目标 | PRD 角色 | 运动类型 | 基准耗时（到核心中点） |
|--------|----------------|------|----------|----------|------------------------|
| 1 | `alert` | U-004 | 警报飞行物 | `linear` 直线 | **30s** |
| 2 | `unknown` | U-002 | 不明飞行物 | `linear` 直线 | **45s** |
| 3 | `warning` | U-003 | 预警飞行物 | `linear` 直线 | **60s** |

**并行示例**：同时点击三个演示按钮 → U-004 约 30s 抵近核心，U-002 约 45s，U-003 约 60s，便于一场放映里依次讲解警报 → 不明 → 预警。

各分段维护独立的 `segmentStartAt[targetId]`；某场 `POST /demo/start` 仅启动对应目标，不影响其他已在飞目标。

### 3.1 背景目标（非预警，始终可见）

不参与分段演示，**全程沿不规则多边形循环飞行**：

| 目标 | 角色 | 运动 |
|------|------|------|
| B-101 | 鸟群 | `polygonLoop` |
| U-001 | 己方 | `polygonLoop` |
| U-005 | 己方 | `polygonLoop` |

### 3.2 分段演示目标初始态

| 目标 | 初始 `visible` | 触发出现 |
|------|----------------|----------|
| U-004 | `false` | `POST /demo/start { segment: "alert" }` |
| U-002 | `false` | `POST /demo/start { segment: "unknown" }` |
| U-003 | `false` | `POST /demo/start { segment: "warning" }` |

### 3.3 直线段到达终点后

- **悬停**在终点（`linear` 完成后 `motionState: 'hovering'`）
- 不消失、不折返
- 处置完成后标记 `disposed`，目标变灰，停止位移

### 3.4 不明目标人工标记与流程切换

U-002（不明）初始走 `unknown` 研判模板；指挥员**手动标记**后切换为对应业务流：

| 标记操作 | `manualOverride` | 研判模板 | 后续流程 |
|----------|------------------|----------|----------|
| 标记警报 | `alert` | `GET /judgment/alert` | 与 U-004 相同：跟踪区可下发、处置闭环 |
| 标记预警 | `warning` | `GET /judgment/warning` | 与 U-003 相同：预警研判与处置规则 |
| 标记白名单 | `friendly` | — | 视为己方，不触发下发 |
| 标记无风险 | `normal` | — | 威胁解除，不触发下发 |

规则：**手动标记优先级高于自动圈层计算**；标记后地图颜色、研判面板、下发按钮态同步切换。

## 4. 全局速度调节

### 4.1 概念

- **基准**：U-004（警报）从起点飞到**核心区域中点** `(0.5, 0.5)` 耗时 **30 秒**（`alertToCoreSec = 30`）
- **倍率** `speedMultiplier`：界面一键调节（如 0.5× / 1× / 1.5× / 2× 滑块或预设按钮）
- **生效规则**：所有与时间相关的量同比缩放

```typescript
const effectiveAlertSec = alertToCoreSec / speedMultiplier;
const effectiveUnknownSec = unknownToCoreSec / speedMultiplier;
const effectiveWarningSec = warningToCoreSec / speedMultiplier;

// 多边形循环
const effectivePolygonSpeed = basePolygonSpeed * speedMultiplier;
```

### 4.2 持久化与 Mock

调节倍率时调用：

```
POST /api/v1/monitoring/demo/timing
Body: { "speedMultiplier": 1.5 }
```

Mock 返回确认；前端 Store 更新倍率，**正在进行的演示立即按新倍率继续**（以调整时刻为新的时间基准，避免位置跳变）。

初始默认值放在 `GET /api/v1/monitoring/scene` 的 `demoTiming` 字段中。

## 5. 运动模型摘要

详见《区域监控-技术设计.md》§3.3，此处为总览：

| type | 适用 | 关键字段 |
|------|------|----------|
| `polygonLoop` | B-101、U-001、U-005 | `vertices[]`, `lapDurationSec` |
| `linear` | U-004、U-002、U-003 | `from`, `to`, `toCoreSec` |

`to` 为直线终点；**到核心中点的用时**由 `toCoreSec` 指定，引擎根据路径长度自动换算线速度。

区域 / 威胁等级：每帧由坐标 + `scene.zones` 计算，不写死在 Mock 里。
